## AI 架构与趋势

### 1) 今日可见的 3 个清晰趋势

**趋势 A：AI Agent 从“框架”走向“产品化客户端/控制台”**
- 代表：`thunderbird/thunderbolt`
- 关键词：多平台、可自托管、可切换模型提供方（本地/私有/前沿模型）、企业可控
- 对 QA 的信号：测试对象从“函数调用”扩展到“端到端体验 + 模型/后端可替换 + 安全合规”。

**趋势 B：Coding Agent 的关键基础设施开始标准化：MCP + 代码检索上下文**
- 代表：`zilliztech/claude-context`
- 关键词：MCP、代码库上下文、检索/embedding、权限边界
- 对 QA 的信号：你们会越来越多地测试“检索质量 + 权限隔离 + 可复现索引”的组合系统，而不是单一模型能力。

**趋势 C：AI 的“输入不确定性”从文本扩展到真实世界信号（Edge / Sensing）**
- 代表：`ruvnet/RuView`
- 关键词：WiFi 信号、DensePose、实时、硬件链路、模型推理与信号处理耦合
- 对 QA 的信号：对“不可控输入”的测试方法（录制回放、数据集基线、稳定性统计）会迁移回 AI Agent 测试：LLM 输出同样是“概率分布”，需要“统计口径 + 观测性”。

## 热门项目：特色与核心优势（结合 Eileen 工作视角）

### 1) thunderbird/thunderbolt
- 链接：https://github.com/thunderbird/thunderbolt
- Topics：ai, ai-agents, on-device-ai, llms

**项目特色**
- “AI You Control”：强调**模型提供方可切换**（本地 / on-prem / 前沿），并尽量减少厂商锁定。
- 多端覆盖：web/iOS/Android/Mac/Linux/Windows，偏“产品级 Agent 客户端/控制台”。

**核心优势（对 Eileen）**
- **混合云/私有化友好**：天然会遇到“同一产品在不同环境接不同模型/不同网络条件”的复杂度，这正是混合云质量保障的主战场。
- **可测性抓手清晰：Provider 抽象层**。只要它把“模型调用/鉴权/限流/流式输出”封装成接口，你就能把不确定性压到最薄。

**对 AI Agent QA 的启发（可落地）**
1. **把模型当成可插拔依赖做 Contract Test**：
   - 统一定义 `Generate()` / `Stream()` / `Embed()`（如有）接口契约：超时、取消、重试、错误码、token 计费/限流行为。
   - 在混合云里常见的坑：代理、DNS、证书、eBPF/sidecar、出口限速；建议把这些变成可自动化的“环境矩阵”。
2. **端到端（Playwright）重点覆盖“模型切换 + 流式输出 + 弱网/断网恢复”**：
   - 关键不是“回答内容一模一样”，而是：UI 不崩、状态可恢复、用户能看懂错误原因。

### 2) zilliztech/claude-context
- 链接：https://github.com/zilliztech/claude-context
- 主要语言：TypeScript
- 关键词：MCP, agentic-rag, code-search, embedding, vector-database

**项目特色**
- 为 Claude Code 提供 code search MCP：让 coding agent 可以把整个代码库当成上下文。
- 本质是一个“**上下文构建服务**”：索引 → 检索 → 返回片段 → 供 Agent 推理。

**核心优势（对 Eileen）**
- 它把“Agent 能力最不稳定的一段（上下文）”拆成了可测组件：
  - **索引是否可复现**（同一 commit + 同一配置应得到相同索引/分片结果）
  - **检索是否可度量**（召回率、命中率、覆盖率、延迟）
  - **权限是否可隔离**（多租户、敏感目录、不同 role 的可见范围）

**对 AI Agent QA 的启发（可落地）**
1. **把 RAG/检索当作“后端服务”来测，不要当作“模型魔法”**：
   - 建一套固定 repo fixture（小型代码库 + 预期命中结果），在 CI 跑检索回归。
2. **安全测试要前移到“检索层”**：
   - prompt injection 里很常见的一类攻击是“引导模型去读不该读的代码/配置”。
   - 你可以在 MCP 层做策略：目录白名单、敏感文件过滤、脱敏；并用 Ginkgo 做策略单测/契约测试。

## 对日常 QA 工作的工程化启发（如何测试此类架构）

下面这部分我会刻意按 Eileen 的“混合云 + AI Agent + Ginkgo/Playwright/K8s”工作方式来写：**优先讲能直接落地的测试设计模式**。

### 1) 先把 AI Agent 的“可测边界”画清楚：三层拆分

1. **工具层（Tool / Function / MCP API）**：最该确定性测试的层
   - 适用：Ginkgo（契约/幂等/权限/错误码）
2. **编排层（Plan/Execute/Retry/Memory/State Machine）**：最该覆盖分支与状态转移的层
   - 适用：Ginkgo（状态机表驱动）、也可做 property-based 测试
3. **模型层（LLM 推理）**：最难确定性测试的层
   - 适用：录制回放 + 小规模在线抽样 + 指标监控（不要只靠单测）

> 关键原则：**把“不可控”尽量压缩到模型层**，并通过回放与评测口径让它“可回归”。

### 2) 混合云/私有化 AI Agent 的典型风险点（建议做成测试清单）

**A. 供应商/部署形态变化引发的兼容问题**（对应 thunderbolt 的“choose your models”）
- 同一接口：不同 Provider 的限流/流式 chunking/错误码可能完全不同
- 建议：定义统一错误码映射 + 适配层 contract test

**B. 权限边界与数据泄露**（对应 claude-context 的“代码检索上下文”）
- 多租户场景最怕：A 租户 prompt 把 B 租户数据“检索出来”
- 建议：
  - 检索层做强隔离（tenant_id 进入索引键空间）
  - 自动化用例覆盖“越权检索/越权工具调用/敏感信息回显”

**C. 可复现性（Reproducibility）**
- 对 Agent：决定性因素不仅是模型版本，还包括：检索数据、索引版本、工具返回、提示词、参数
- 建议：把每次执行产出 `trace.json`，包含：
  - prompt hash / model id / temperature
  - tool calls（入参/出参摘要）
  - retrieval hits（doc id + score）
  - 关键决策点（plan steps）

### 3) 后端（Ginkgo）可直接套用的“工具 API 合同测试”骨架

> 目标：把 Agent 的动作空间变成 **可控、可断言、可回归** 的 API。

```go
package tool_contract_test

import (
  "bytes"
  "encoding/json"
  "net/http"

  "github.com/onsi/ginkgo/v2"
  . "github.com/onsi/gomega"
)

type ToolResp struct {
  Code    string                 `json:"code"`
  Message string                 `json:"message"`
  Data    map[string]any         `json:"data"`
  TraceID string                 `json:"trace_id"`
}

var _ = ginkgo.Describe("Tool API Contract", func() {
  ginkgo.It("should be idempotent for same request_id", func() {
    payload := map[string]any{
      "request_id": "e2e-001",
      "input":      map[string]any{"q": "ping"},
    }
    b, _ := json.Marshal(payload)

    call := func() ToolResp {
      req, _ := http.NewRequest("POST", "http://localhost:8080/api/tools/search", bytes.NewReader(b))
      req.Header.Set("Content-Type", "application/json")
      resp, err := http.DefaultClient.Do(req)
      Expect(err).ToNot(HaveOccurred())
      defer resp.Body.Close()

      Expect(resp.StatusCode).To(Equal(http.StatusOK))
      var tr ToolResp
      _ = json.NewDecoder(resp.Body).Decode(&tr)
      Expect(tr.TraceID).ToNot(BeEmpty())
      return tr
    }

    r1 := call()
    r2 := call()

    Expect(r1.Code).To(Equal("OK"))
    Expect(r2.Code).To(Equal("OK"))
    // 重点：幂等性断言（按你们业务字段调整）
    Expect(r2.Data).To(Equal(r1.Data))
  })
})
```

### 4) 端到端（Playwright）建议重点测“流式输出 + 可恢复性 + 人能看懂的错误”

```ts
import { test, expect } from '@playwright/test';

test('Agent chat: streaming + provider switch should be recoverable', async ({ page }) => {
  await page.goto('https://your-console.example.com');

  // 1) 切换模型/Provider（核心是“配置可生效 + UI 状态一致”）
  await page.getByRole('button', { name: '模型设置' }).click();
  await page.getByRole('combobox', { name: 'Provider' }).selectOption('onprem-llm');
  await page.getByRole('button', { name: '保存' }).click();

  // 2) 发起一次任务
  await page.getByRole('textbox', { name: '输入' }).fill('请帮我总结本次发布的风险点，并给出回归建议');
  await page.getByRole('button', { name: '发送' }).click();

  // 3) 对流式输出：不追求逐字一致，追求“最终完成 + 结构可解析”
  const msg = page.getByTestId('assistant-message').last();
  await expect(msg).toContainText('回归');

  // 4) 失败场景：模拟断网/超时（按产品能力替换）
  // 这里建议你们提供一个“重试”按钮，并断言重试后 trace_id 变化且任务能恢复
});
```

## 可落地的行动指南（如何在现有自动化框架中应用）

下面给出一套“1~2 周内可以落地”的最小闭环，贴合 Eileen 的技术栈：Ginkgo（后端）+ Playwright（E2E）+ K8s（混合云部署）。

### 第 1 步：在仓库里建立统一的 `agent-trace` 与回放资产
1. 新建目录：`qa/agent_quality/`
2. 统一落盘：每次 Agent 执行输出 `trace.json`（建议最少包含：model/provider、prompt hash、tool calls、retrieval hits、最终结果摘要）
3. 录制回放：
   - 工具 API：支持 mock（根据 request_id 回放固定响应）
   - 检索：支持固定索引快照或固定 top-k 命中

### 第 2 步：Ginkgo 做三类“确定性”后端测试（高 ROI）
- **Contract**：OpenAPI / JSON Schema 校验
- **Idempotency**：同一 `request_id` 重试不会产生副作用
- **Auth/RBAC**：租户隔离、敏感工具（读配置/查库）必须有权限边界

### 第 3 步：Playwright 固化 5 条关键路径回放（优先做少而精）
1. 登录 → 创建会话 → 发送任务 → 流式输出完成
2. 切换 Provider/模型 → 任务成功
3. 弱网/断网 → UI 可恢复（重试/继续）
4. 工具调用失败 → 错误提示一致且可定位（带 trace_id）
5. 多轮对话记忆 → “可控记忆”不会泄露跨租户数据

### 第 4 步：K8s 环境矩阵（混合云友好）让问题提前暴露
- 最小矩阵建议：
  - `kind`（本地）+ `staging`（云）+ `onprem`（私有）
- 每个环境至少覆盖：
  - DNS/证书/代理
  - 限流/超时（Ingress、LB、Sidecar）
  - HPA 扩缩容下的会话一致性（sticky session / state store）

### 第 5 步：引入“统计口径”的自动化质量闸门（适配 LLM 不确定性）
- 对评测集（10~50 条即可）计算并做回归：
  - 任务完成率
  - 工具调用成功率
  - 平均/95 分位延迟
  - 拒答率（以及拒答原因分类）
- 产出差分报告：对比上一个版本，**关注指标漂移**而不是逐字差异。
