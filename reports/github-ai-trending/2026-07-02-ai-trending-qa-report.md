## 对日常 QA 工作的工程化启发

### 1. Agent 测试要用 E2E 场景承载单点断言

后续用例不建议单独写“工具 API 是否返回 200”这类孤立验证。更好的组织方式是：先设计真实用户任务，再把 API contract、权限、幂等、重试、trace、输出 schema 等校验放进步骤的中间状态或最终验证点。

示例场景：用户提交一次“安全扫描并生成修复建议”任务。

1. 用户在控制台创建扫描任务，并选择目标应用。
2. 系统生成任务计划，展示将调用的工具、权限范围和预计耗时。
3. Agent 执行探测，过程中写入 trace、工具输入输出摘要和失败重试记录。
4. 系统输出漏洞列表、证据、修复建议和风险等级。
5. 用户确认一条修复建议后，系统生成变更草案或工单。

验证点应包括：权限越界被拒绝、工具调用 schema 合法、重复提交不产生重复任务、扫描失败可重试、最终报告可追溯到原始证据。

### 2. Golang Ginkgo 更适合守住后端确定性边界

对于 OmniRoute、Vibe-Trading 这类平台型或高风险 Agent，后端测试应优先覆盖：

- Provider/工具 API contract：请求、响应、错误码、超时、重试策略稳定。
- 幂等性：同一个 trace_id 或 business_id 重放时，不重复扣费、不重复下单、不重复创建任务。
- 权限边界：只读 connector 不能升级成写操作，安全工具不能访问未授权域名或文件。
- 审计完整性：计划、工具调用、用户确认、失败恢复都能按 trace 串起来。

### 3. Playwright 要覆盖用户可感知的长链路

Agent 产品的前端自动化不应只检查按钮是否可点击，而要覆盖用户可见的状态变化：任务已创建、执行中、可取消、失败可重试、结果可下载、审计可查看。对于流式输出或长任务，可以采用“最终一致性 + 关键中间态”的断言方式，避免把模型 token 级输出写死。

### 4. 数据链路需要像代码一样纳入质量门禁

exercises-dataset 这类项目提醒我们：AI 产品质量并不只由模型决定。训练/评测数据、媒体资源、多语言字段、分类标签都会影响最终体验。建议在现有自动化框架里增加数据质量检查：字段完整性、枚举合法性、资源可访问性、重复记录、异常值、跨语言字段缺失。

## 测开行动建议

1. **建立 `ai_agent_quality/` 场景库：** 按“真实用户任务”组织 E2E 用例，每个场景包含输入、工具链路、期望中间态、最终验收点和可回放快照。
2. **统一 trace 规范：** 所有 Agent 任务必须有 trace_id，串联计划、工具调用、模型响应、用户确认、错误恢复和审计记录。
3. **把工具调用转成 contract test：** 为每个工具 API 维护 JSON Schema、错误码矩阵、权限矩阵和幂等性用例。
4. **为 prompt/模型/provider 变更建立差分回归：** prompt、模型版本、检索策略、工具列表变化时，自动触发回放集并生成差分报告。
5. **补齐高风险 Agent 的安全门禁：** 对交易、安全扫描、文件操作、外部消息发送等场景，默认覆盖只读边界、人工确认、kill switch 和审计。
6. **沉淀 Playwright 长任务模板：** 覆盖任务创建、状态轮询、取消/重试、结果渲染、弱网恢复和刷新后状态恢复。

2. 用户确认执行。
   - 预期中间态：工具调用符合 schema；越权工具被拒绝；事件流持续写入。
3. Agent 执行过程中出现一次可恢复失败。
   - 预期中间态：系统按策略重试；重试记录进入审计日志；用户可见状态不丢失。
4. Agent 输出最终结果。
   - ✅ 最终验证点：结果结构化、证据可追溯、状态为 completed、重复刷新不改变结果。
5. 用户下载报告或创建后续工单。
   - ✅ 最终验证点：导出内容与页面结果一致，工单关联 trace_id。
