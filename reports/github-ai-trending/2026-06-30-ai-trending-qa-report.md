## 今日结论

今天的 AI Trending 项目呈现出两个明显方向：一类是 **Agent 工作流产品化**，把多角色、多工具、多步骤任务包装成可交付流程；另一类是 **AI 应用基础设施**，围绕身份认证、权限边界、媒体处理、投研流程、OSINT 信息聚合等具体场景落地。

对资深测试开发工程师来说，最有价值的启发不是“再补几个单点 API 校验”，而是把这些项目抽象成真实用户链路：**用户触发目标 -> Agent/工具编排 -> 外部系统交互 -> 结果生成 -> 可观测审计 -> 异常恢复**。Ginkgo 后端自动化和 Playwright E2E 应该共同服务这条链路：后端负责契约、权限、幂等、状态机与 trace；前端负责关键路径回放、流式输出、错误态、弱网与最终可见结果。

## 对日常测试开发工作的启发

### 1. E2E 用例要覆盖完整业务链路

今天这些项目共同说明：AI 产品的质量风险往往不在单个接口，而在跨步骤协作中暴露。例如 Agent 分工正确但工具权限过宽，检索结果正确但最终报告没有引用，视频任务启动成功但导出失败后没有恢复入口。

因此，测试用例应默认按真实用户场景组织：

- **Agent 编排链路：** 目标输入 -> 计划生成 -> 工具调用 -> 中间状态可观测 -> 结果交付 -> 审计记录。
- **权限链路：** 登录 -> 租户/角色切换 -> 工具授权 -> 越权拦截 -> 操作日志。
- **长任务链路：** 上传/创建任务 -> 多阶段处理 -> 进度展示 -> 失败重试 -> 最终产物验收。
- **证据链路：** 任务输入 -> 多源检索 -> 证据筛选 -> 结论生成 -> 引用与可解释性检查。

### 2. Ginkgo 后端自动化要从 API 断言升级为状态机断言

后端测试不能只检查 HTTP 200 和字段存在。更适合的做法是把一个 Agent 任务视作状态机：`created -> planning -> tool_calling -> aggregating -> completed / failed / cancelled`。每个状态都应有明确的事件、trace_id、错误码、幂等规则和可重试边界。

建议新增以下 Ginkgo 场景：

```go
var _ = Describe("Agent task E2E contract", func() {
  It("should complete a tool-backed task with auditable trace", func() {
    // 1. 创建任务：断言 task_id / trace_id / 初始状态
    // 2. 模拟工具返回：断言状态迁移与事件顺序
    // 3. 查询结果：断言结构化输出、引用、审计日志
    // 4. 重放同一请求：断言幂等与重复提交保护
  })
})
```

### 3. Playwright E2E 要把“用户可见状态”作为最终验证点

AI 应用常见问题是后端状态正确，但前端没有可理解的反馈。Playwright 场景应把用户真正看到的内容纳入断言：流式输出开始/结束、工具调用卡片、错误提示、重试入口、引用来源、导出产物、审计记录入口。

建议优先沉淀三类路径：

1. **关键成功路径：** 登录 -> 创建任务 -> 查看流式进度 -> 获得结果 -> 查看引用/日志。
2. **受控失败路径：** 工具超时 -> 页面提示可重试 -> 用户重试 -> 状态恢复 -> 最终成功或明确失败。
3. **权限拦截路径：** 低权限用户触发高风险工具 -> 被拦截 -> 页面展示原因 -> 审计中可查询。

## 今日可复用测试设计模板

```text
用例名称：Agent 工具链路端到端验证
前置条件：用户已登录；测试环境启用固定模型/Mock Provider；工具权限按角色配置完成
执行步骤：
1. 用户输入一个需要调用外部工具的真实任务。
   预期中间状态：页面展示任务已创建；后端生成 task_id 与 trace_id。
2. Agent 进入规划阶段并选择工具。
   预期中间状态：事件流包含 planning 与 tool_selected；工具参数满足 schema。
3. 工具执行并返回结果。
   预期中间状态：后端记录工具耗时、状态码、错误码或成功结果；失败时进入可重试分支。
4. Agent 聚合结果并生成最终回答。
   ✅ 最终验证点：用户能看到结构化结论、引用/证据、任务日志入口；重复提交不会产生重复副作用。
```
